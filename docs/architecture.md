# Розподіл частин проєкту Orest

## Frontend

Frontend-роль виконує інтерфейс Telegram-чату, тобто те, що бачить користувач:

- команди `/start`, `/about`, `/help`;
- текстові відповіді бота;
- реакція на звичайні повідомлення;
- повідомлення про невідому команду.

Чому це frontend: саме через Telegram-клієнт користувач взаємодіє з ботом. Користувач не бачить Python-код, Docker або змінні середовища, а бачить тільки чат, команди й відповіді.

Схематично:

```text
Користувач -> Telegram chat interface -> бот
```

## Backend

Backend у цьому проєкті - це Python-застосунок, який містить логіку роботи бота.

До backend належать:

- `app/main.py`;
- `app/dev.py`;
- обробники команд `start()`, `about()`, `help_command()`;
- обробник текстових повідомлень `echo()`;
- обробник невідомих команд `unknown_command()`;
- обробник помилок `error_handler()`;
- налаштування логування;
- завантаження `.env`;
- робота з Telegram Bot API через бібліотеку `python-telegram-bot`.

Чому це backend: саме ця частина приймає події від Telegram, обробляє їх, приймає рішення і надсилає відповідь користувачу.

Також до backend-інфраструктури можна віднести:

- `requirements.txt`;
- `Dockerfile`;
- `docker-compose.yml`;
- `.env.example`;
- `scripts/scan-secrets.ps1`.

Ці файли не є бізнес-логікою бота, але допомагають запускати, налаштовувати й підтримувати backend.

## Database

У поточній версії проєкту окремої бази даних немає.

Чому database немає: після перезапуску бот не має власної пам'яті про попередні взаємодії. Він просто отримує повідомлення, обробляє його і одразу відповідає.

## API

У цьому проєкті API можна розглядати на кількох рівнях.

### 1. API бота для користувача

Це команди й поведінка, через які користувач взаємодіє з ботом:

```text
/start
/about
/help
звичайне текстове повідомлення
невідома команда
```

У коді це відповідає:

```text
/start -> start()
/about -> about()
/help -> help_command()
text message -> echo()
unknown command -> unknown_command()
```

Чому це API: це публічний інтерфейс бота. Користувач знає, які команди можна надіслати і яку поведінку очікувати у відповідь.

### 2. Зовнішній Telegram Bot API

Проєкт працює через Telegram Bot API. Саме через нього бот отримує оновлення від Telegram і надсилає повідомлення назад у чат.

Бібліотека `python-telegram-bot` не є API цього проєкту, але вона є SDK (набір функцій для програмування), тобто Python-інтерфейсом до зовнішнього Telegram Bot API.

Схематично:

```text
Telegram Bot API
        ↓
python-telegram-bot
        ↓
app/main.py
```

### 3. Конфігураційне API

Проєкт також має конфігураційний інтерфейс через змінні середовища:

```text
BOT_TOKEN
LOG_LEVEL
```

`BOT_TOKEN` потрібен для підключення бота до Telegram.

`LOG_LEVEL` керує рівнем логування, наприклад `INFO`, `DEBUG`, `ERROR`.

Чому це API: через ці значення можна змінювати поведінку запуску проєкту без зміни коду.

## Підсумок

```text
Frontend  -> Telegram-чат, команди й повідомлення, які бачить користувач.
Backend   -> Python-код бота, логіка обробки повідомлень і запуск застосунку.
Database  -> у поточній версії відсутня.
API       -> команди бота, Telegram Bot API і конфігураційні змінні.
```

У підсумку: користувач взаємодіє з ботом через Telegram як frontend, Python-застосунок обробляє цю взаємодію як backend, база даних поки не використовується, а API складається з команд бота, зовнішнього Telegram Bot API та змінних конфігурації.
